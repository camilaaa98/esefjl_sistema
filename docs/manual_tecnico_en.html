1. General Architecture and Technology Stack

The SISFARMA PRO ecosystem has been structured under a "Robust Simplicity" philosophy. The system allows efficient execution on both local servers (WAMP) and distributed cloud infrastructures, optimizing performance under limited hardware conditions.

Core Technology Stack:

  • Processing: PHP 8.2+ with optimized Zend engine.
  • Persistence: Data Abstraction Layer (PDO) compatible with SQLite and PostgreSQL.
  • Frontend: "Native-Web" architecture (ES2022 + CSS Custom Properties). No heavy external dependencies to ensure instant loading in limited connectivity zones (2G/3G).
  • Visual Identity: Design system based on medical color psychology (Forest Green & Ivory White).

2. Model-View-Controller (MVC) Pattern Implementation

To ensure software maintainability, a strict separation of concerns has been implemented following the MVC pattern.

// Example File Distribution /core /Models -> (Data Abstraction: CRUD) /Controllers -> (Business Logic and Validation) Database.php -> (Singleton Pattern for Connection) ViewHelper.php -> (Static Helper for UI) /views -> (Clean HTML templates with minimal logic) /assets -> (Segregated CSS/JS static assets)

Information Flow:

  1. The user interacts with a View (e.g., registro_entrega.php).
  2. The request is captured by a Controller (DeliveryController.php), which validates session permissions and sanitizes input.
  3. The controller invokes the Model (DeliveryModel.php) to persist the transaction in the DB.
  4. The result is returned to the view, dynamically formatted using the ViewHelper.

3. Data Dictionary and Relational Model

The database schema is normalized to avoid redundancy and ensure the integrity of patient and medication records.

Primary Entities:

TableResponsibilityPrimary Key
usuarios Stores credentials (bcrypt hashes) and roles. id (PK)
sedes Defines municipal IPS and administrative branches. id (PK)
productos Master catalog of medications (Generic names). id (PK)
inventario Transactional relationship: Product + Branch + Stock + Expiration. id (PK)
pacientes Enrolled citizens directory (SIAU). documento (Unique)
entregas Audit log of each dispatch performed. id (PK)

Transactional Logic:

Each medication output recorded in the entregas table triggers a logical process (trigger or repository method) that updates the stock in the inventario table, guaranteeing the integrity of balances by sede_id.

4. Industrial Cybersecurity Protocols

Given the sensitivity of the clinical data handled by ESE Fabio Jaramillo, proactive security layers have been integrated.

  • SQLi Protection: Total shielding through Prepared Statements in all persistence layers.
  • Password Hashing: Continuous use of password_hash() with the BCRYPT algorithm, the industry's most robust standard.
  • XSS Sanitization: Mandatory filtering of user inputs before rendering in views.
  • RBAC (Role Based Access Control): Privilege verification at the server level for each action controller.
Security Warning: Never modify configuration files (core/config.php) in production environments without an active SSL connection.

5. Deployment, Migration, and Scalability Guide

SISFARMA PRO is prepared to grow at the same pace as ESE Fabio Jaramillo.

WAMP Deployment Procedure:

  1. Enable the headers_module and rewrite_module in Apache.
  2. Ensure the PHP version is 8.2.x for return type compatibility.
  3. Configure the docroot pointing to the project's root folder.

Cloud Migration (PostgreSQL / Supabase):

To scale to a distributed database:

  1. Update the Database.php class by changing the connection DSN from SQLite to PostgreSQL.
  2. Execute the migration SQL script included in the /sql folder.
  3. The system will automatically adapt queries thanks to the PDO layer.

6. Maintenance and Backup Protocol

Information integrity is the organization's most valuable asset.

Backup Policies:

  • Cold Copy: Daily backup of the core/esefjl.db file (If using SQLite).
  • Index Optimization: Monthly execution of VACUUM to compact the database and improve response times in audit reports.
  • Error Logs: Constant monitoring of the error file to identify failure patterns in the municipal IPS network.
