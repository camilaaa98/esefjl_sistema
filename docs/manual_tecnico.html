1. Arquitectura General y Stack Tecnológico

EL ecosistema Farmacia ESEFJL ha sido diseñado bajo una filosofía de "Simplicidad Robusta", permitiendo su ejecución tanto en servidores locales WAMP de baja potencia como en infraestructuras cloud distribuidas.

Core Tecnológico:

  • Procesamiento: PHP 8.2+ con motor Zend optimizado.
  • Persistencia: Capa de abstracción de datos (PDO) compatible con SQLite y PostgreSQL.
  • Frontend: Arquitectura "Native-Web" (ES2022 + CSS Custom Properties). Sin dependencias externas pesadas para garantizar carga instantánea en zonas de conectividad limitada (2G/3G).
  • Identidad Visual: Sistema de diseño basado en la psicología del color médico (Forest Green & Ivory White).

2. Implementación del Patrón Modelo-Vista-Controlador (MVC)

Para garantizar la mantenibilidad del software, se ha implementado una separación de responsabilidades estricta siguiendo el patrón MVC.

// Ejemplo de Distribución de Archivos /core /Models -> (Abstracción de Datos: CRUD) /Controllers -> (Lógica de Negocio y Validación) Database.php -> (Singleton Pattern para Conexión) ViewHelper.php -> (Helper estático para UI) /views -> (Templates HTML limpios con lógica mínima) /assets -> (Activos estáticos CSS/JS segregados)

Flujo de Información:

  1. El usuario interactúa con una Vista (Ej: registro_entrega.php).
  2. La petición es capturada por un Controlador (DeliveryController.php), el cual valida permisos de sesión y sanitiza la entrada.
  3. El controlador invoca al Modelo (DeliveryModel.php) para persistir la transacción en la DB.
  4. El resultado es devuelto a la vista, formateado dinámicamente mediante el ViewHelper.

3. Diccionario de Datos y Modelo Relacional

El esquema de base de datos está normalizado para evitar la redundancia y garantizar la integridad de los registros de pacientes y medicamentos.

Entidades Principales:

TablaResponsabilidadLlave Primaria
usuarios Almacena credenciales (hashes bcrypt) y roles. id (PK)
sedes Define las IPS municipales y sedes administrativas. id (PK)
productos Catálogo maestro de medicamentos (Nombres genéricos). id (PK)
inventario Relación transaccional: Producto + Sede + Stock + Vencimiento. id (PK)
pacientes Directorio de ciudadanos vinculados (SIAU). documento (Unique)
entregas Log de auditoría de cada despacho realizado. id (PK)

Lógica Transaccíonar:

Cada entrega de medicamento (entregas) dispara un trigger lógico que descuenta el stock de la tabla inventario filtrando por la sede_id de la sesión activa.

4. Protocolos de Ciberseguridad Industrial

Dada la sensibilidad de los datos clínicos manejados por la ESE Fabio Jaramillo, se han integrado capas de seguridad proactiva.

  • Protección SQLi: Blindaje total mediante Prepared Statements en todas las capas de persistencia.
  • Hashing de Password: Uso de password_hash() con algoritmo BCRYPT, el estándar más robusto de la industria.
  • Sanitización XSS: Filtrado mandatorio de entradas de usuario antes de la renderización en vistas.
  • RBAC (Role Based Access Control): Verificación de privilegios a nivel de servidor en cada controlador de acción.
Advertencia de Seguridad: Nunca modifique los archivos de configuración (core/config.php) en entornos de producción sin una conexión SSL activa.

5. Guía de Despliegue, Migración y Escalabilidad

Farmacia ESEFJL está preparado para crecer al mismo ritmo que la ESE Fabio Jaramillo.

Procedimiento de Despliegue en WAMP:

  1. Habilitar el módulo headers_module y rewrite_module en Apache.
  2. Asegurar que la versión de PHP sea 8.2.x para compatibilidad con el tipado de retornos.
  3. Configurar el docroot apuntando a la carpeta raíz del proyecto.

Migración a la Nube (PostgreSQL / Supabase):

Para escalar a una base de datos distribuida:

  1. Actualizar la clase Database.php cambiando el DSN de conexión de SQLite a PostgreSQL.
  2. Ejecutar el script SQL de migración incluido en la carpeta /sql.
  3. El sistema automáticamente adaptará las consultas gracias a la capa PDO.

6. Protocolo de Mantenimiento y Backups

La integridad de la información es el activo más valioso de la organización.

Políticas de Respaldo:

  • Copia Fría: Respaldo diario del archivo core/esefjl.db (En caso de usar SQLite).
  • Optimización de Índices: Ejecución mensual de VACUUM para compactar la base de datos y mejorar tiempos de respuesta en reportes de auditoría.
  • Logs de Error: Monitoreo constante del archivo de errores para identificar patrones de fallas en la red de las IPS municipales.
